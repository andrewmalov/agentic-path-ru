Самая большая ошибка новичков в агентной инженерии: запрос слишком много за раз.

## Почему декомпозиция важна

Агенты лучше всего работают с ограниченными, чёткими задачами. Когда вы запрашиваете слишком много:

- Контекст переполняется и ранние детали теряются
- Ошибки накапливаются, когда последующие шаги строятся на ранних ошибках
- Восстановление после неудач тратит всё предыдущее

Маленькие задачи означают быстрый фидбек, проще отладка, лучшие результаты.

## Правило трёх частей

Перед подсказкой спросите: Могу ли я описать это в трёх частях?

1. **Что** — Конкретный результат, который я хочу
2. **Где** — Какие файлы/функции изменять
3. **Ограничения** — Что не изменять

Если вы не можете сформулировать всё три, задача, скорее всего, слишком большая.

## Стратегии декомпозиции

### Вертикальные нарезки

Режьте по пути фич, а не по техническим слоям.

**Вместо:**

- "Создать схему базы данных"
- "Построить API эндпоинты"
- "Создать фронтенд формы"

**Попробуйте:**

- "Создать поток создания пользователя: схему, эндпоинт и базовую форму"
- "Добавить редактирование пользователя: эндпоинт обновления и форму"
- "Добавить удаление пользователя: эндпоинт с подтверждением"

Каждый слой должен быть завершён и протестирован.

### Порядок зависимостей

Когда задачи имеют зависимости, делайте их явными:

1. Определить типы модели данных (без внешних зависимостей)
2. Построить слой хранения (зависит от типов)
3. Создать API (зависит от хранения)
4. Построить UI (зависит от API)

Каждый шаг должен работать изолированно, прежде чем переходить к следующему.

## Оценка размера задач

**Слишком маленькая:** "Добавить точку с запятой на строку 47" — быстрее сделать самому

**Слишком большая:** "Построить систему аутентификации" — слишком много решений

**Идеальная:** "Создать форму входа, которая постит на /api/auth/login и сохраняет токен в localStorage" — чёткая область, проверяемый результат

## Шаблон подсказки

```
Задача: [Что вы хотите сделать]

Контекст:
- [Т relevantное фона]
- [П relevant files или паттерны, которым следовать]

Ограничения:
- [Что не менять]
- [Какие паттерны следовать]

Критерии успеха:
- [Как вы поймёте, что сделано]
```

## Валидация результатов

Результат агента выглядит правдоподобно. Вот опасность. Обращайтесь с кодом агента как с кодом талантливого младшего: скорее всего работает для happy path, может пропустить edge cases, может не следовать вашим конвенциям.

### Чеклист валидации

**1. Решает ли она задачу?**
Прочитайте код — не просто запустите его. Она решает реальное требование, а не просто вашу подсказку?

**2. Проверьте границы:**

- Пустые входы
- Null/undefined значения
- Граничные условия (off-by-one, макс значения)
- Ошибочные случаи
- Параллельный доступ

**3. Ищите галлюцинации:**

- API методов, которые не существуют
- Параметров, которые не работают как предполагается
- Функций, вызываемых с неправильными сигнатурами

Если что-то незнакомо, проверьте, что существует.

**4. Безопасность:**

- Есть ли валидация входов?
- Нет SQL/команд инъекций?
- Проверяется auth/authorization?
- Чувствительные данные не логируются?
- Нет hardcoded secrets?

**5. Стиль и конвенции:**

- Следует существующим паттернам?
- Имена понятные и согласованные?
- Сообщения об ошибках полезные?

### Когда отклонять

**Отклоняйте когда:**

- Подход фундаментально неправилен (даже если работает)
- Требуется слишком много рефакторинга для соответствия стандартам
- Проблемы безопаности, требующие redesign

**Принимайте с изменениями когда:**

- Логика звучит, но нужен стильовый adjustment
- Требуется небольшое обработка edge case

**Принимайте как есть когда:**

- Выполняет требования, обрабатывает грани, следует конвенциям, проходит проверку безопасности
