Реализация — это то, где большинство инженеров впервые сталкиваются с агентами. Успех зависит от подхода, а не только от промптов.

## Высокоценные задачи

**Генерация шаблонов** — CRUD операции, scaffolding API, форм-компонентов, DTO, конфигов. Агенты справляются быстро и надёжно. Здесь заявления о "10x" почти соответствуют правде.

**Реализация функций** — Лучше всего работает, когда требования ясны, паттерны существуют в кодовой базе, и объём ограничен (один PR, несколько файлов).

**Исправление багов** — Чёткие баги — отличные задачи для агентов. "User login fails with null pointer when email contains '+'" лучше чем "исправь поток авторизации."

**Рефакторинг** — Механический рефакторинг идеален: переименование по кодовой базе, выделение функции/класса, конверсация паттернов. Сложное перестроение сложнее.

**Трансформации данных** — migration скрипты, конверсия форматов, ETL логика. Чёткие входы и выходы делают эти задачи дружелюбными для агентов.

## Рабочий процесс

1. **Планирование** — Знать, какие файлы изменятся, желаемый конечный результат, и что НЕ должно меняться.

2. **Настройка контекста** — Предоставить релевантный код, существующие паттерны для следования, и ограничения.

3. **Генерация** — Для сложных задач, спросить сначала о плане: "Опиши, как ты бы реализовал [функцию]. Пока не пиши код."

4. **Валидация** — Работает ли? Обрабатывает edge cases? Следует ли конвенциям? Есть ли проблемы безопасности?

5. **Доработка** — "Это не обрабатывает случай, когда..." или "Следуй паттерну в [файл] вместо этого."

## Что тормозит

- **Перекрысложенные промпты** — Начинать просто, добавлять детали по мере необходимости
- **Недоограниченные запросы** — "Build the feature" оставляет слишком много решений агенту
- **Борьба с агентом** — После 3+ повторных запросов без прогресса, переосмыслить или сделать самому
- **Недостаточный контекст** — Пропуск контекста ведёт к некорректному выводу
- **Неправильный инструмент** — Некоторый код быстрее писать вручную

## Шаблоны промптов

**Реализация функции:**

```
Реализуй [функцию] в [файл/модуль].

Требования:
- [Конкретное требование 1]
- [Конкретное требование 2]

Следуй паттерну, использованному в [существующий пример].
Не модифицируй [сохраняемые области].
```

**Исправление бага:**

```
Баг: [описание]
Воспроизведение: [шаги или код]
Ожидается: [поведение]
Фактически: [поведение]

Исправь это в [файл]. Корень проблемы: [если известен].
```

**Рефакторинг:**

```
Рефакторинг [модуль/функции] для [желаемое изменение].

Сохранить публичный API без изменений.
Поддерживать всю существующую функциональность.
[Дополнительные ограничения]
```
