CI/CD конвейеры — это конфигурационно-тяжеленый сектор, хорошо подходящий для агентов. Мониторинг и обслуживание — это постоянные задачи, где агенты помогают с setup, реагированием на инциденты и рутинными задачами.

## CI/CD и деплой

### Где помогают агенты

**Конфигурация конвейеров:** GitHub Actions, GitLab CI, Jenkins, CircleCI. Агенты знают распространённые паттерны и адаптируются под ваш стек.

**Infrastructure as Code:** Terraform, CloudFormation, Pulumi — генерация ресурсов, модификация существующей инфраструктуры, создание переиспользуемых модулей.

**Автоматизация деплоя:** Dockerfiles, Kubernetes манифесты, Helm чарты, deploy скрипты, процедуры отката.

**Конфигурация окружений:** Переменные окружения, управление секретами, конфигурационные файлы.

### На что обращать внимание

**Безопасность** — CI/CD — это security-critical зона. Агент-генерируемые конвейеры могут раскрывать секреты в логах, использовать перегруженные права доступа или пропускать security сканирование. Всегда проводите security-ревью CI/CD изменений.

**Особенности вендоров** — Каждый CI систем имеет свои нюансы. Тщательно тестируйте сгенерированные конвейеры.

**Состояная ресурсов** — Инфрастранаправления могут вызвать удаление, простои, финансовые сюрпризы или потерю данных. Используйте паттерн plan/apply.

### Шаблоны промптов

**Генерация конвейера:**

```
Создай GitHub Actions workflow для [язык/фреймворк] проекта.

Требования:
- Запускать тесты на PR
- Деплоить в [окружение] при слиянии в main
- Использовать [специфичные сервисы/инструменты]

Следовать security best practices.
```

**Инфраструктура:**

```
Сгенерировать Terraform для [тип ресурса] с:
- [Конкретные требования]
- [Ограничения]

Следовать паттернам в [существующий файл/модуль].
```

**Dockerfile:**

```
Создай Dockerfile для [тип приложения].

Требования:
- Multi-stage build
- Минимальный финальный образ
- Запускать от non-root
- [Прочие требования]
```

## Мониторинг и обслуживание

### Настройка мониторинга

**Конфигурация алертов:** Prometheus правила, CloudWatch alarms, Datadog мониторы. Агенты понимают распространённые паттерны того, что нужно мониторить.

**Создание дашбордов:** Grafana дашборды, Kibana визуализации. Описываете что хотите увидеть — получаете стартовую конфигурацию.

**Агрегация логов:** Лог парсинг, поисковые запросы, обнаружение аномалий.

### Реагирование на инциденты

**Анализ корневой причины:** "Вот ошибка и недавние изменения. Что может вызвать это? Что нужно проверить?"

**Отладка:** Анализировать логи, интерпретировать стек трейсы, трассировать потоки запросов, выявлять паттерны ошибок.

**Черчение постмортема:** "Помоги написать постмортем для [инцидент]. Включить: summary, timeline, корневую причину, влияние, remediation, lessons learned."

### Рутинные задачи

- **Обновление зависимостей** — Анализировать версии, выявлять breaking changes, генерировать update PR
- **Технический долг** — Выявлять паттерны улучшений, генерировать refactoring план
- **Оптимизация производительности** — Анализировать проблемы, предлагать подходы, генерировать бенчмарки
- **Поддержка безопасности** — Remediation уязвимостей, применение патчей, hardening конфигураций

### На что обращать внимание

- **Alert fatigue** — Сгенерированные алерты могут быть слишком чувствительны или плохо калиброваны. Настраивайте на основе реального опыта.
- **Dashboard overload** — Больше дашбордов — не значит лучше. Спрашивайте: Какое решение информирует каждый панель?
- **Maintenance scope creep** — Ограничивайте задачи реальной проблемой, which решается.

### Шаблоны промптов

**Правила алертов:**

```
Создай Prometheus alerting rules для [тип сервиса].

Мониторить:
- Error rate exceeding [порог]
- Latency above [порог]
- Resource utilization above [порог]

Включить подходящие уровни серьёзности и аннотации.
```

**Расследование инцидента:**

```
Вижу [ошибка/симптом] в продакшене.

Релевантный контекст:
- [Недавние изменения]
- [Error логи]
- [Метрики]

Что может быть причиной? Что проверить в первую очередь?
```

**Runbook:**

```
Создай runbook для обработки [тип инцидента].

Включить:
- Detection (как мы знаем, что это происходит)
- Triage (как оценить серьёзность)
- Mitigation (немедленные действия)
- Resolution (полное исправление)
- Follow-up (пост-инцидент)
```
