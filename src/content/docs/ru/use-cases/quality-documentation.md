Тестирование и документация — высокоценные сценарии для агентов. Тесты самовалидируются — вы сразу знаете, работают ли они. Документация постоянно недостаточно поддерживается — агенты резко снижают трение.

## Тестирование

### Где агенты выделяются

**Генерация юнит-тестов** — Данная функция, агенты генерируют исчерпывающие тесты: happy path, edge cases (null, empty, границы), error conditions. Это, вероятно, единственное применение, где "10x" заявления соответствуют реальности.

**Выявление тестовых случаев** — "Какие тестовые случаи нужно учесть для этой функции? Сфокусируйся на edge cases и error conditions." Вы часто найдёте случаи, о которых не думали.

**Генерация моков и фикстур** — Тестовые данные структуры, мок реализации, фабричные функции. Скучная setup работа идеальна для агентов.

**Регрессионные тесты** — "Создай тест, который поймал бы этот баг: [опиши баг]."

### На что обращать внимание

- **Тесты, которые проходят по неправильной причине** — Тесты, которые мокируют проверяемое, или утверждения, которые всегда проходят
- **Галлюцинации утверждений** — Агенты могут утверждать поведение, не соответствующее реальности
- **Копипаст тестов** — Похожие тесты, которые не добавляют значимого покрытия

Всегда проверяйте, что тесты могут падать where они должны.

### Тест-first рабочий процесс

1. **Пишем или генерируем тесты первыми:** "Имея это требование [опиши], напиши тесты, которые проверяют корректную реализацию."
2. **Проверяем и дорабатываем:** Убеждаемся, что тесты захватывают то, что мы хотим.
3. **Генерируем реализацию:** "Вот тесты. Напиши код, который делает их проходящими."
4. **Проверяем всё:** Запускаем тесты, проверяем реализацию, ищем пробелы.

Это даёт лучшие результаты, чем реализация первая, потому что тесты определяют контракт.

### Шаблоны промптов

**Юнит-тесты:**

```
Напиши юнит-тесты для этой функции:
[вставь функцию]

Покрыть:
- Нормальная работа
- Edge случаи (пустой ввод, null, границы)
- Error условия

Использовать [тест фреймворк]. Следовать конвенциям в [пример файле].
```

**Регрессионный тест:**

```
Баг исправлен: [описание]
Корневая причина: [объяснение]
Fix: [кратко]

Напиши тест, который поймает этот баг, если он вернётся.
```

## Документация

### Типы и подходы

**Документация кода:** Инлайн комменты (объясняют _зачем_), функциональная документация, класс/модуль docs. Агенты отлично справляются, потому что читают код и выводят, что нужно документировать.

**API документация:** Endpoints, SDKs, integration guides. Генерация из кода, OpenAPI спеков, или примеров.

**Архитектурная документация:** Систем overviews, decision records, диаграммы. Нужен больший человеческий ввод, но агенты помогают с генерацией.

### На что обращать внимание

- **Галлюцинации функций** — API, которых не существует, параметры, которые не работают. Проверяйте все заявления против реального кода.
- **Пропущенный контекст** — Почему принимались решения, gotchas, производительные соображения. Добавляйте вручную — часто самые ценные.
- **Риск устаревания** — Easy to generate ≠ easy to maintain. Стройте процессы обновления, not just generation.

### Шаблоны промптов

**README bootstrap:**

```
Создай README для этого проекта.

Проект: [опиши проект]
Тех стек: [список технологий]
Целевая аудитория: [кто будет использовать]

Включить: установка, quick start, базовое использование, contributing.
```

**API документация:**

```
Сгенерировать API документацию для этого endpoint:

[вставь endpoint код или OpenAPI spec]

Включить: описание, параметры, запрос/ответ примеры, error коды.
Формат как Markdown.
```

**ADR:**

```
Помоги написать ADR по выбору [опция].

Контекст: [опиши ситуацию]
Решение: [что решили]
Последствия: [что это значит]

Включить альтернативы, рассматриваемые.
```
