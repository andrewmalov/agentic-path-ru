Копируйте и адаптируйте эти шаблоны под свои нужды. Каждый шаблон включает ключевые компоненты для эффективных промптов.

## Промпты для реализации

### Новая функция

```
Реализуй [название функции].

## Требования
- [Функциональное требование 1]
- [Функциональное требование 2]
- [Нефункциональное требование]

## Технический контекст
- Язык/Фреймворк: [указать]
- Релевантные файлы: [список файлов]
- Паттерн для следования: [ссылка на файл или описание]

## Ограничения
- Не модифицировать: [сохраняемые области]
- Сохранить: [требования backward compatibility]
- Требование производительности: [если есть]

## Критерии приёмки
- [ ] [Критерий 1]
- [ ] [Критерий 2]
- [ ] [Критерий 3]
```

### API endpoint

```
Создай API endpoint для [цель].

## Спецификация endpoint
- Метод: [GET/POST/PUT/DELETE]
- Путь: [/api/v1/...]
- Аутентификация: [required/optional/none]

## Request
- Тело: [опиши структуру или пример]
- Параметры: [query params, path params]
- Заголовки: [требуемые заголовки]

## Response
- Успех (200): [опиши/пример]
- Ошибки: [список с кодами статуса]

## Детали реализации
- Локация handler: [путь к файлу]
- Сервис: [название сервиса]
- Следовать паттерну в: [пример файл]
```

### Database migration

```
Создай database migration для [цель].

## Требуемые изменения
- [Изменение 1: добавить таблицу/колонку/индекс]
- [Изменение 2]
- [Изменение 3]

## Схема
[Опиши таблицы, колонки, типы, ограничения]

## Учёты
- Существующие данные: [как обрабатывать]
- Обратимость: [требования к rollback]
- Производительность: [большие таблицы]

## Фреймворк миграций: [Prisma/Knex/etc.]
```

## Промпты для исправления багов

### С известной причиной

```
Исправь баг: [краткое описание]

## Воспроизведение
1. [Шаг 1]
2. [Шаг 2]
3. [Шаг 3]

## Текущее поведение
[Что происходит сейчас]

## Ожидаемое поведение
[Что должно происходить]

## Корневая причина
[Твой анализ того, почему это происходит]

## Ограничения
- Минимизировать изменения
- Не модифицировать [защищённые области]
- Добавить регрессионный тест
```

### Требуется расследование

```
Debug issue: [описание симптома]

## Контекст
- Когда происходит: [условия]
- Сообщение об ошибке: [если есть]
- Релевантные логи: [вставь релевантные логи]
- Недавние изменения: [что изменилось недавно]

## Что я уже пробовал
- [Попытка 1 и результат]
- [Попытка 2 и результат]

## Вопросы
- Что может быть причиной?
- Как проверить корневую причину?
- Какой самый безопасный fix?
```

## Промпты для тестирования

### Юнит тесты

```
Напиши юнит-тесты для [function/class name] в [file].

## Тест фреймворк: [Jest/pytest/etc.]

## Требования покрытия
- Нормальные случаи работы
- Граничные случаи:
  - Пустой/null ввод
  - Граничные значения
  - [Доменно-специфичные границы]
- Ошибочные условия:
  - [Ошибочный случай 1]
  - [Ошибочный случай 2]

## Стиль
- Следовать паттернам в [пример тест файла]
- Использовать [describe/it или naming convention]
- Замокать [что мокать]
```

### Интеграционные тесты

```
Напиши интеграционные тесты для [компонент/функция].

## Объём тестирования
- Тестировать взаимодействие между [компонент A] и [компонент B]
- Использовать [реальный/замоканный] [база данных/внешний сервис]

## Сценарии
1. [Happy path сценарий]
2. [Error сценарий]
3. [Edge case сценарий]

## Требования к setup
- [Prerequisites]
- [Требуемые тест данные]

## Фреймворк: [указать]
```

## Промпты для рефакторинга

### Выделить функцию/класс

```
Рефакторинг: Выдели [описание] в [функцию/класс].

## Текущее расположение кода: [file:строки]

## Желаемый результат
- Новое имя [функции/класса]: [название]
- Новое расположение: [file]
- Параметры: [что должно принимать]
- Возвращаемое значение: [что должно возвращать]

## Ограничения
- Сохранить существующее поведение
- Обновить всех вызывающих
- Поддерживать backward compatibility
```

## Промпты для code review

### Пред-commit проверка

```
Проверь этот код перед commit.

## Код
[вставь код]

## Контекст
- Purpose: [что делает этот код]
- Part of: [большая фича/PR]

## Специфичные опасения
- [Область 1, которую не уверен]
- [Область 2, которую хочу обсудить]

## Проверить
- Баги и логические ошибки
- Security проблемы
- Performance соображения
- Code стиль
- Граничные случаи
```

## Советы по использованию шаблонов

1. **Заполняйте детали** - Заменяйте все placeholder в квадратных скобках
2. **Предоставляйте контекст** - Больше контекста = лучший вывод
3. **Будьте явны об ограничениях** - Что НЕ делать так же важно, как что делать
4. **Включайте примеры** - Показывайте паттерны для следования
5. **Итерируйтесь** - Первый ответ редко идеален; дорабатывайте follow-up
