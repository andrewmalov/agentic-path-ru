CI/CD пайплайны — это в основном конфигурация, и они хорошо подходят для агентов. Мониторинг и поддержка — это постоянные задачи, где агенты помогают с настройкой, реагированием на инциденты и рутинными задачами.

## CI/CD и развёртывание

### Где помогают агенты

**Конфигурация пайплайнов:** GitHub Actions, GitLab CI, Jenkins, CircleCI. Агенты знают распространённые паттерны и адаптируются к вашему стеку.

**Инфраструктура как код:** Terraform, CloudFormation, Pulumi — генерируйте ресурсы, модифицируйте существующую инфраструктуру, создавайте переиспользуемые модули.

**Автоматизация развёртывания:** Dockerfile, Kubernetes-манифесты, Helm-чарты, скрипты деплоя, процедуры отката.

**Конфигурация окружения:** Переменные окружения, управление секретами, конфигурационные файлы.

### На что обращать внимание

**Безопасность** — CI/CD критичен для безопасности. Сгенерированные агентами пайплайны могут раскрывать секреты в логах, использовать излишне разрешительные права или пропускать сканирование безопасности. Всегда проводите security-ревью изменений CI/CD.

**Особенности вендоров** — Каждая CI-система имеет различия. Тщательно тестируйте сгенерированные пайплайны.

**Stateful-ресурсы** — Изменения инфраструктуры могут вызвать удаление, простой, неожиданные расходы или потерю данных. Используйте паттерны plan/apply.

### Паттерны промптов

**Генерация пайплайна:**

```
Создайте GitHub Actions workflow для проекта на [язык/фреймворк].

Требования:
- Запускать тесты на PR
- Деплоить в [окружение] при merge в main
- Использовать [конкретные сервисы/инструменты]

Следуйте лучшим практикам безопасности.
```

**Инфраструктура:**

```
Сгенерируйте Terraform для [тип ресурса] с:
- [Конкретные требования]
- [Ограничения]

Следуйте паттернам в [существующий файл/модуль].
```

**Dockerfile:**

```
Создайте Dockerfile для [тип приложения].

Требования:
- Multi-stage build
- Минимальный финальный образ
- Запуск от не-root пользователя
- [Другие требования]
```

## Мониторинг и поддержка

### Настройка мониторинга

**Конфигурация алертинга:** Правила Prometheus, CloudWatch-алерты, Datadog-мониторы. Агенты понимают распространённые паттерны того, что мониторить.

**Создание дашбордов:** Grafana-дашборды, визуализации Kibana. Опишите, что хотите видеть; получите стартовую конфигурацию.

**Агрегация логов:** Правила парсинга, поисковые запросы, обнаружение аномалий.

### Реагирование на инциденты

**Анализ первопричины:** "Вот ошибка и недавние изменения. Что могло это вызвать? Что проверить?"

**Помощь в отладке:** Анализ логов, интерпретация стектрейсов, трассировка потоков запросов, выявление паттернов ошибок.

**Написание постмортемов:** "Помогите написать постмортем для [инцидент]. Включите: резюме, таймлайн, первопричину, влияние, исправление, уроки."

### Задачи поддержки

- **Обновление зависимостей** — Анализ версий, выявление breaking changes, генерация PR для обновлений
- **Очистка технического долга** — Выявление паттернов улучшений, генерация планов рефакторинга
- **Оптимизация производительности** — Анализ проблем, предложение подходов, генерация бенчмарков
- **Поддержка безопасности** — Исправление уязвимостей, применение патчей, харденинг конфигурации

### На что обращать внимание

- **Alert fatigue** — Сгенерированные алерты могут быть слишком чувствительными или плохо откалиброванными. Настраивайте на основе реального опыта.
- **Переизбыток дашбордов** — Больше дашбордов — не значит лучше. Спрашивайте: Какое решение информирует каждая панель?
- **Разрастание области поддержки** — Ограничивайте задачи реальной решаемой проблемой.

### Паттерны промптов

**Правила алертинга:**

```
Создайте правила алертинга Prometheus для [тип сервиса].

Мониторить:
- Превышение порога ошибок [порог]
- Задержка выше [порог]
- Использование ресурсов выше [порог]

Включите соответствующие уровни серьёзности и аннотации.
```

**Расследование инцидента:**

```
В продакшене наблюдаю [ошибка/симптом].

Релевантный контекст:
- [Недавние изменения]
- [Логи ошибок]
- [Метрики]

Что могло это вызвать? Что проверить в первую очередь?
```

**Runbook:**

```
Создайте runbook для обработки [тип инцидента].

Включите:
- Детекция (как мы узнаём, что это происходит)
- Триаж (как оценить серьёзность)
- Смягчение (немедленные действия)
- Разрешение (полное исправление)
- Follow-up (послеинцидентные действия)
```
