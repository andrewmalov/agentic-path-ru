Тестирование и документация — высокоценные кейсы использования агентов. Тесты самоаттестуются — вы сразу знаете, работают ли они. Документация постоянно недостаточно поддерживается — агенты значительно снижают трение.

## Тестирование

### Где агенты преуспевают

**Генерация юнит-тестов** — Дан функцию, агенты генерируют комплексные тесты: happy path, граничные случаи (null, пустота, границы), условия ошибок. Это, вероятно, самый высокоокупаемый вариант применения для большинства команд.

**Выявление тест-кейсов** — "Какие тест-кейсы следует рассмотреть для этой функции? Сфокусируйтесь на граничных случаях и условиях ошибок." Вы часто обнаруживаете случаи, которые не рассматривали.

**Генерация моков и фикстур** — Тестовые структуры данных, мок-реализации, фабричные функции. Утомительная настройка — идеальная задача для агентов.

**Регрессионные тесты** — "Создайте тест, который поймал бы этот баг: [описание бага]"

### На что обращать внимание

- **Тесты, которые проходят по неправильным причинам** — Тесты, которые мокают то, что тестируется, или имеют assertions, которые всегда проходят
- **Hallucinated assertions** — Агенты могут утверждать поведение, которое не соответствует реальности
- **Копипаста тесты** — Похожие тесты, которые не добавляют осмысленного покрытия

Всегда проверяйте, что тесты могут падать, когда должны.

### Тест-первый рабочий процесс

1. **Напишите или сгенерируйте тесты первыми:** "Учитывая это требование [описать], напишите тесты, которые проверят правильную реализацию."
2. **Просмотрите и уточните:** Убедитесь, что тесты захватывают то, что вы действительно хотите.
3. **Генерируйте реализацию:** "Вот тесты. Напишите код, который их проходит."
4. **Верифицируйте всё:** Запустите тесты, просмотрите реализацию, проверьте пробелы.

Это даёт лучшие результаты, чем реализация-первый подход, потому что тесты определяют контракт.

### Паттерны промптов

**Юнит-тесты:**

```
Напишите юнит-тесты для этой функции:
[вставьте функцию]

Покройте:
- Нормальную работу
- Граничные случаи (пустой ввод, null, границы)
- Условия ошибок

Используйте [тестовый фреймворк]. Следуйте соглашениям в [пример файла].
```

**Регрессионный тест:**

```
Исправленный баг: [описание]
Корневая причина: [объяснение]
Исправление: [краткое описание]

Напишите тест, который поймает этот баг, если он вернётся.
```

## Документация

### Типы и подходы

**Документация кода:** Inline-комментарии (объясняйте _почему_), docs для функций/методов, классов/модулей. Агенты преуспевают здесь, потому что читают код и делают вывод о том, что нужно документировать.

**API-документация:** Эндпоинты, SDK, руководства по интеграции. Генерируйте из кода, OpenAPI-спецификаций или примеров.

**Архитектурная документация:** Обзоры систем, records решений, диаграммы. Требуется больше человеческого руководства, но агенты помогают с генерацией.

### На что обращать внимание

- **Hallucinated features** — API, которые не существуют, параметры, которые не работают. Проверяйте все утверждения против фактического кода.
- **Отсутствующий контекст** — Почему были приняты решения, подводные камни, соображения производительности. Добавляйте вручную — они часто наиболее ценны.
- **Риск устаревания** — Легко сгенерировать ≠ легко поддерживать. Стройте процессы обновления, а не только генерации.

### Паттерны промптов

**README с нуля:**

```
Создайте README для этого проекта.

Проект: [описание проекта]
Tech stack: [список технологий]
Целевая аудитория: [кто будет это использовать]

Включите: установку, быстрый старт, базовое использование,_contributing.
```

**API-документация:**

```
Сгенерируйте API-документацию для этого эндпоинта:

[вставьте код эндпоинта или OpenAPI-спецификацию]

Включите: описание, параметры, примеры запроса/ответа, коды ошибок.
Форматируйте как Markdown.
```

**ADR:**

```
Помогите написать ADR для выбора [опция].

Контекст: [описание ситуации]
Решение: [что мы решили]
Последствия: [что это значит]

Включите рассмотренные альтернативы.
```
